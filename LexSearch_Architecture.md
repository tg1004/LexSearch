# LexSearch — Complete Architecture & Product Specification
## AI-Powered Semantic Search Engine for Indian Legal Judgements

---

## 1. PRODUCT OVERVIEW

### What You Are Building
LexSearch is a semantic search engine for Indian legal judgements. Users type natural language questions like "What are the grounds for bail in murder cases?" and get AI-synthesized answers with citations pointing to real Supreme Court and High Court judgements.

### Core Value Proposition
- Normal search finds keywords. LexSearch finds meaning.
- Every answer is grounded in real documents with clickable citations.
- No hallucination risk — the AI only summarizes what it retrieves.

### Target Users
- Law students researching case law
- Lawyers looking for precedents quickly
- Journalists researching legal topics
- General public understanding their legal rights

- Top retrieved chunks sent to LLM with query

## 7. LLM FALLBACK CHAIN — COMPLETE IMPLEMENTATION SPEC

### How It Works
1. User sends search query with optional preferred_provider
2. System builds RAG prompt (query + retrieved chunks)
3. LLMService tries providers in this order:
   - If preferred_provider set: try that first, then others
   - If auto: try Groq → Gemini → Ollama
4. On success: return LLMResponse with provider_used field
5. On failure (rate limit, network error, timeout): log error, try next
6. If all fail: return structured error response to frontend
7. Frontend shows "Service temporarily unavailable, try again"

### Failure Detection
- Groq rate limit: HTTP 429 response
- Groq any error: any Exception from groq SDK
- Gemini rate limit: HTTP 429 or google.api_core.exceptions.ResourceExhausted
- Ollama unavailable: ConnectionRefusedError or requests.exceptions.ConnectionError
- All providers: catch-all Exception with logging

### Provider Availability Check (for UI selector)
- Groq: assume available (check on actual call)
- Gemini: assume available (check on actual call)
- Ollama: ping localhost:11434/api/tags with 2 second timeout

```python
def hybrid_search(query: str, embedding: list, filters: dict) -> list[Chunk]:
    # Run both searches in parallel (asyncio.gather)
    vector_results = qdrant_search(embedding, filters, top_k=20)
    keyword_results = elasticsearch_search(query, filters, top_k=20)
    
    # Merge with Reciprocal Rank Fusion
    # RRF score = sum of 1/(rank + 60) for each result's rank in each list
    merged = reciprocal_rank_fusion(vector_results, keyword_results)
    
    return merged[:10]  # top 10 chunks
```

### Step 3: Prompt Building
```python
SYSTEM_PROMPT = """
You are a legal research assistant specializing in Indian law.
Answer the user's question using ONLY the provided document excerpts.
Do not use any external knowledge.

Rules:
1. Cite every claim using [1], [2], [3] notation matching the source numbers
2. If the excerpts don't contain enough information, say so clearly
3. Write in clear, plain English — not legalese
4. Be concise — 3 to 5 sentences maximum for the main answer
5. Never invent case names, dates, or legal principles not in the excerpts
"""

def build_rag_prompt(query: str, chunks: list[Chunk]) -> str:

{sources}
```

## 9. CHUNKING STRATEGY

### Why Chunking Matters
- Too large: embedding loses specific details, retrieval is imprecise
- Too small: chunks lose context, answers are incomplete

### Strategy: Recursive Character Splitting with Overlap
```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,          # ~800 characters per chunk
    chunk_overlap=150,       # 150 char overlap between chunks
    separators=[
        "\n\n",              # paragraph breaks first
        "\n",                # then line breaks
        ". ",                # then sentence endings
        " "                  # last resort: word boundaries
    ]
)
```

### Overlap Explained
If a sentence spans the boundary between two chunks, the overlap ensures it appears fully in at least one chunk. Without overlap you lose the sentence that straddles the boundary.

## 14. IMPORTANT IMPLEMENTATION NOTES FOR THE LLM BUILDING THIS

1. Use async/await throughout the FastAPI backend — all database calls, all HTTP calls, all LLM calls must be async

2. The LLM fallback chain must catch ALL exceptions, not just specific ones — use broad Exception catch as the last resort

3. Run vector search and keyword search in parallel using asyncio.gather — do not run them sequentially

4. Always validate that Qdrant and Elasticsearch return results before passing to LLM — if both return empty, return "no results found" without calling LLM

5. The citation parser must be robust — LLM sometimes returns [1] and sometimes returns (1) or "Source 1" — handle all formats

6. Cache search results in Redis with a 1 hour TTL — same query should not hit the LLM twice

7. The sentence-transformers model downloads on first run (~90MB) — add a startup event in FastAPI to pre-load it so first search is not slow

8. For the highlighted text in DocumentViewer — store the chunk start/end character position during ingestion in Qdrant payload so you can highlight without re-searching

9. Rate limit the /api/search endpoint to 20 requests per minute per IP using slowapi — prevents abuse on free tier deployments

10. All API endpoints except /api/health, /api/auth/login, /api/auth/signup, and /api/providers require JWT authentication

11. The admin dashboard endpoints must additionally check is_admin flag in JWT — do not rely only on authentication

12. Use connection pooling for PostgreSQL (SQLAlchemy pool_size=10) — do not create a new connection per request

13. Elasticsearch index mapping must be defined explicitly before indexing — do not rely on auto-mapping as it will get field types wrong for dates and arrays
